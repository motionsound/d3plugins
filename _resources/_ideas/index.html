1. Clip thumbnails recipe works render is intermittent

A small frame-0 still per video, via the LIVE ThumbnailSystem instance. The recipe is correct and has returned real frames for un-previewed assets — BUT it's intermittent: some sessions the engine's thumbnail render never runs and every call returns the 128×128 "?" placeholder (even with retries). Pulled from video_countdowns on 2026-06-01 until we understand what makes the engine's thumbnail pump run (GPU load? a trigger?).

The recipe (Python via /api/session/python/execute):

import d3
ts = d3.subsystems.getSystem(d3.ThumbnailSystem)   # the LIVE engine instance (NOT a fresh ThumbnailSystem())
f  = _ex.evaluateFromString("transportmanager:<T>").track.findLayerByName("<layer>").module.video.file
tex = ts.getThumbnail(f, ts.getThumbnailSize(f), 0)   # frame 0 @ native size. WHEN it works: real frame instantly.
# tex is a DxTexture; None or 128x128 "?" = not ready / placeholder.
tex.download(True); tex.saveFile(path)                # DxTexture.saveFile writes a PNG
# read bytes -> base64 -> data URI in the page:  img.src = "data:image/png;base64," + b64
# detect placeholder by PNG dims: real = (w,h) != (128,128)  [IHDR: struct.unpack('>I', b[16:20]) / b[20:24]]
# NOTE: only frame 0 / native size is reliable. Other frames + non-native sizes render async (placeholder first).
# renderPreview(Rect,Colour,bool,bool) draws to SCREEN (returns None) — not a capturable texture.

2. Master fade state — feedOutputMode not API-readable

Goal: show "INACTIVE — FADED DOWN / HOLD" when the project's master output is faded. disguise's OSC custom-monitoring exposes feedOutputMode = "Fade up" / "Fade down" / "Output hold" (r30.7+), but like currentFeatureName it's an OSC/expression-bar variable, NOT on the API objects we can reach. Found only actions (guisystem.feed.startFadeDown, onHoldOutput, fadeCol) and d3.d3.state (a D3State with enableBlackout but no fade-up/hold/down enum). The statusOf() in video_countdowns already has the hooks (M.fadeState === 'down'|'hold') — just need a readable source. Possible revive: route feedOutputMode via OSC-out → a device variable → read over LiveUpdate (command-bus style).

Blocked by the same thing as #1: arbitrary-frame extraction (3rd arg of getThumbnail) only reliably returns frame 0; other frames render async/placeholder, and there's no found path to a full-res on-demand decode. Revisit alongside #1.

4. Multi-transport render binding disguise limitation

Render-state (module.frameNumberDisplay / tSecElapsed) only populates for ONE transport at a time (an internal binding latched to whichever transport was accessed first; survives editor switches and even single-transport mode). guisystem.currentTransportManager is read-only; transport.activate() doesn't switch it; no setter found. video_countdowns sidesteps this: it shows ALL transports and uses each transport's own tCurrent — so Layer/Section clocks work everywhere, only the Clip method + mute are limited to the bound transport. Revive idea: if a real "make this transport active for render" command is ever found, add a button.
